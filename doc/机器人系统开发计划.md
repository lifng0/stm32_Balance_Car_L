# 机器人系统开发计划

更新时间：2026-05-30

## 1. 项目目标

以当前 `STM32` 平衡小车工程为唯一开发基线，完成《进阶式挑战性综合项目II课题任务书》中与本项目硬件条件匹配的核心任务，最终交付一台可独立演示的机器人小车系统。

最终目标按以下原则执行：

- 以“能稳定演示、能按要求答辩、能快速交付”为第一优先级。
- 机械臂相关功能不纳入本项目范围。
- 由于实际使用的是 `K210` 而不是深度相机，因此所有原本依赖深度相机完成的视觉识别/跟随类任务，改由 `K210` 提供目标检测、颜色跟踪、二维码识别等视觉能力。
- “建图”与“雷达避障/追踪”主要由 `T-mini Plus` 激光雷达承担，不强行让 `K210` 去做不适合的深度建图。
- 不为“看起来完整”而额外堆砌与交付无关的复杂模块，优先做老师明确要求且可稳定演示的功能。

## 2. 当前硬件基线

### 2.1 已确认保留的现有硬件

- `STM32F103` 平衡小车底盘
- 电机驱动、编码器、`MPU6050`
- `OLED`
- 蓝牙模块
- `K210` 视觉模块

### 2.2 新增硬件

- `树莓派5 8GB`
- `YDLIDAR T-mini Plus`
- 降压稳压模块
  - 作用：给树莓派提供稳定电压/电流，避免直接由电池供电导致掉压、重启、串口异常

### 2.3 当前不再作为项目目标的老扩展件

- 超声波
- `CCD`
- 电磁巡线
- 四路巡线
- 手柄模块

## 3. 任务书要求到本项目的落地解释

结合任务书和当前真实硬件，最终需要落地的功能清单如下。

### 3.1 必做能力

- 树莓派5开发环境搭建
- `ROS2` 开发环境搭建
- `Docker` 容器化环境搭建
- `STM32` 与树莓派5协同控制
- 底盘运动控制
- 传感器数据采集与处理
- 模块驱动打通
- `K210` 视频/识别相关功能接入
- 颜色跟踪
- 目标跟踪
- 二维建图
- 雷达避障
- 雷达追踪
- 遥控/自主模式切换
- 最终整车独立演示

### 3.2 明确不做

- 机械臂控制
- 多舵机抓取闭环
- 为了“像 ROSMASTER”而强行复刻其全部机械结构与全向运动学
- 未被老师要求、但会显著拉长周期的额外功能
  - 例如复杂 GUI、云端平台、大型数据库、完整导航竞赛系统

### 3.3 重要设计取舍

- 当前车体是“平衡小车”，不是麦克纳姆底盘。
  - 因此项目重点放在“机器人系统设计与实现、感知、协同控制、演示功能闭环”。
  - 不以“严格复刻全向底盘运动学”作为交付条件。
- `K210` 替代深度相机时，主要承担“视觉识别与目标方向估计”。
  - 不要求 `K210` 单独完成真实深度恢复或稠密建图。
- “建图”使用 `T-mini Plus + ROS2` 完成。
- “目标跟踪/颜色跟踪”使用 `K210` 完成。
- 若演示中需要目标距离，则采用“`K210` 给方向，激光雷达给距离”的融合方案，而不是做伪深度。

## 4. 总体技术方案

## 4.1 系统分层

### `STM32` 侧职责

- 自平衡控制主循环
- 电机驱动、编码器、姿态解算、PID
- `OLED` 状态显示
- 蓝牙遥控兜底
- `K210` 串口接收与本地快速响应
- 与树莓派5的底层通信接口
- 安全保护
  - 倾倒保护
  - 低压保护
  - 通信超时保护

### 树莓派5侧职责

- `ROS2` 运行环境
- `Docker` 容器环境
- `T-mini Plus` 驱动与数据处理
- 建图与可视化
- 高层任务调度
- 自主模式逻辑
- 与 `STM32` 的控制/状态同步
- 可选日志与演示脚本

### `K210` 侧职责

- 颜色目标识别
- 目标框位置输出
- 二维码识别
- 简单目标类别识别

## 4.2 推荐连接关系

- `K210 <-> STM32`
  - 保持现有 `USART2` 连接不动，优先复用现有代码与串口协议
- `调试口`
  - `USART1` 默认继续作为 `STM32` 调试/日志口使用，不作为树莓派主通信口
- `树莓派5 <-> STM32`
  - 优先使用原本预留给雷达的 `UART3 / USART3`，通过 `TTL转USB` 模块接入树莓派 `USB` 口，建立新的主控协同链路
  - `STM32` 原有蓝牙在 `UART5`，可保留为手动兜底控制
- `T-mini Plus <-> 树莓派5`
  - 优先直接接树莓派5，而不是再接回 `STM32`
  - 原因：后续 `ROS2` 建图、雷达可视化、数据处理都更顺，且不占用 `STM32` 计算资源
  - 结合 `T-mini Plus` 手册，雷达本体接口为 `TTL UART`，供电为 `5V`，通信电平为 `3.3V`
  - 推荐优先使用官方/配套 `USB 转 UART` 转接板接入树莓派 `USB`
  - 不推荐默认把雷达重新挂回 `STM32 USART3`
    - 该串口已改作 `STM32 <-> 树莓派5` 主通信口
    - 即使硬接到 `STM32`，后续也仍需要再转发到树莓派，链路更长、调试更慢
- 电池 -> 降压稳压模块 -> 树莓派5
  - 必须先做供电稳定性验证，再进入软件联调

## 4.3 通信策略

采用“低层实时控制在 STM32，高层任务调度在树莓派”的双主协同架构。

### `STM32 -> 树莓派`

- 姿态/速度/电池/模式状态
- 关键事件
  - 倾倒
  - 启停
  - `K210` 识别结果

### 树莓派 -> `STM32`

- 模式切换命令
- 速度/转向目标
- 启停命令
- 自主任务指令

### 通信协议原则

- 第一版不要做复杂协议栈
- 先做稳定的串口帧协议即可
- 必备字段：
  - 帧头
  - 长度
  - 命令字
  - 载荷
  - 校验
- 必备机制：
  - 心跳
  - 超时失联保护
  - 命令确认

## 5. 开发阶段规划

## 阶段0：目标冻结与验收定义

### 目标

在正式开发前把“必须做什么、明确不做什么、怎样算完成”一次讲清楚。

### 任务

- 固化最终功能清单
- 固化硬件连接方案
- 固化每项功能的验收动作
- 固化演示顺序

### 输出

- 本开发计划文档
- 一份最终验收清单

### 验收标准

- 所有人对“交付边界”理解一致
- 后续不再随意引入额外功能

## 阶段1：底盘基线稳定化

### 目标

确保当前 `STM32` 平衡车基线稳定，作为后续所有功能的可靠底座。

### 任务

- 确认当前裁剪后的工程可稳定编译、烧录、起立
- 复测保留模式：
  - 标准模式
  - 负重模式
  - `K210` 二维码
  - `K210` 视觉巡线
  - `K210` 跟随
  - `K210` 自主学习
  - `K210` 数字识别
- 整理 `STM32` 侧当前串口资源分配
- 预留树莓派5通信接口

### 设计思路

- 这一阶段不做 ROS2，不接入雷达
- 只确保底盘与现有 `K210` 功能稳定

### 输出

- 稳定版 `STM32` 固件基线
- 串口资源表

## 阶段2：树莓派5基础环境搭建

### 目标

让树莓派5尽快具备后续开发条件。

### 任务

- 烧录系统并完成基础配置
- 验证供电稳定性
- 配置网络、SSH、串口权限、时区、自动启动
- 安装 Docker
- 安装 `ROS2`
- 建立基础工作区
- 记录环境搭建脚本

### 设计思路

- 优先脚本化，不手工重复配置
- 如果原生环境和 Docker 并存，Docker 作为最终标准开发环境

### 输出

- 树莓派5环境初始化脚本
- Docker 基础镜像或 Dockerfile
- `ROS2` 工作区骨架

### 并行性

- 可与阶段1后半段并行

## 阶段3：`STM32` 与树莓派5通信打通

### 目标

建立“树莓派发命令、STM32 执行动作、再回传状态”的最小闭环。

### 任务

- 设计并实现串口帧协议第一版
- `STM32` 新增上位机命令解析模块
- 在树莓派5侧打通 `TTL转USB` 串口链路
- 树莓派5编写串口通信节点
- 实现最小命令集合
  - 查询状态
  - 启动/停止
  - 设置模式
  - 设置 `Move_X / Move_Z`
- 实现通信心跳与失联保护

### 设计思路

- 第一版只做最小指令，不做复杂抽象
- 优先让树莓派能可靠控制底盘

### 输出

- 串口协议说明
- `STM32` 通信固件
- `ROS2` 串口桥节点

### 验收动作

- 树莓派下发前进、后退、转向命令，小车稳定响应
- 树莓派掉线时小车自动进入安全状态

## 阶段4：`T-mini Plus` 接入与雷达能力恢复

### 目标

让项目重新获得“距离感知、建图、雷达避障、雷达追踪”能力，但这次放到树莓派5/ROS2主线上完成，而不是回退到旧的 `STM32` 雷达分支。

### 任务

- 接入 `T-mini Plus`
- 固化雷达物理接入方案
  - 首选 `T-mini Plus -> USB 转 UART 转接板 -> 树莓派 USB`
  - 备选 `T-mini Plus TTL UART -> 树莓派 GPIO UART`
- 打通驱动与数据可视化
- 输出激光扫描数据到 `ROS2`
- 验证静态测距与动态采样
- 设计雷达前向障碍物检测逻辑
- 实现最小避障模式
- 实现最小追踪模式

### 设计思路

- 不恢复旧的 `STM32` 雷达模式代码
- 直接基于树莓派5侧完成雷达功能，减少重复开发
- `STM32` 只接受运动目标，不做重雷达计算
- 雷达默认走树莓派侧 `USB` 接入，尽量不占用树莓派主串口资源
- 若后续必须走 GPIO 串口，再单独处理串口使能、权限和抗干扰问题

### 输出

- 雷达驱动节点
- 雷达避障节点
- 雷达追踪节点

### 验收动作

- 可实时查看雷达点云/扫描
- 小车在前方有障碍时能停止或绕避
- 对选定目标可实现简化追踪

## 阶段5：`K210` 功能统一接入

### 目标

让 `K210` 不再只是局部演示模块，而是成为整车可调度的视觉感知模块。

### 任务

- 盘点现有 `K210` 串口输出协议
- 统一 `K210` 识别结果格式
- `STM32` 对 `K210` 数据做标准化封装
- 将 `K210` 状态转发给树莓派
- 明确各视觉模式的最小交付版本
  - 二维码识别
  - 颜色跟踪
  - 目标跟踪
  - 数字识别

### 设计思路

- 先复用现有 `STM32 + K210` 能力
- 再把关键信息上送给树莓派
- 不急着把 `K210` 直接迁到树莓派侧，除非后续确实成为瓶颈

### 输出

- `K210` 数据协议文档
- `STM32 -> Pi5` 视觉状态上报链路

### 验收动作

- 树莓派能够读到 `K210` 的识别结果
- 可以在高层模式中根据视觉结果切换行为

## 阶段6：建图能力实现

### 目标

完成任务书里的“建图”要求。

### 任务

- 基于 `T-mini Plus` 完成二维建图链路
- 完成里程或姿态辅助接口
- 在 `ROS2` 中实现地图生成与保存
- 形成固定演示流程

### 设计思路

- 建图以“可稳定生成 2D 地图并保存展示”为验收标准
- 不默认引入完整导航栈，除非老师后续明确要求
- 若无高质量轮式里程计，则优先接受“手动遥控建图 + 可视化保存”的方案

### 输出

- 地图生成流程
- 地图保存脚本
- 建图演示说明

### 验收动作

- 能输出一张可展示的环境地图
- 可在答辩中解释建图数据链路

## 阶段7：自主模式整合

### 目标

把各单点能力整合成最终可演示的自主模式。

### 任务

- 统一模式管理
  - 手动蓝牙模式
  - 树莓派接管模式
  - 雷达避障模式
  - 雷达追踪模式
  - 视觉跟踪模式
- 增加模式切换保护
- 增加 `OLED` 状态提示
- 定义演示流程中的一键启动顺序

### 设计思路

- 模式切换必须显式
- 先保证模式之间不打架，再追求“自动无缝切换”
- 任何自主模式下都保留紧急停车能力

### 输出

- 最终模式状态机
- 模式切换协议

### 验收动作

- 手动/自主可稳定切换
- 切换后行为与预期一致

## 阶段8：最终联调与演示封装

### 目标

把系统从“能跑功能”提升到“能独立演示”。

### 任务

- 梳理上电顺序
- 梳理容器启动顺序
- 编写一键启动脚本
- 编写演示脚本
- 编写异常恢复说明
- 编写项目报告素材清单

### 设计思路

- 交付的不是一堆模块，而是一套“别人按步骤就能演示”的系统
- 演示流程要短、稳、可重复

### 输出

- 一键启动脚本
- 演示手册
- 联调记录
- 答辩素材清单

## 8. 最短交付路径

如果目标是“尽快形成可演示版本”，最短路径如下：

1. 让 `STM32` 基线稳定可演示
2. 打通树莓派5与 `STM32` 通信
3. 接入 `T-mini Plus` 并实现最小避障
4. 接入 `K210` 并保留最核心视觉模式
5. 完成二维建图
6. 完成手动/自主模式切换

这条路径上，以下内容后置：

- 高级优化
- 大规模重构
- 非关键模式的细节打磨
- 可视化界面美化

## 9. 关键风险与应对

### 风险1：树莓派供电不稳

- 表现：随机重启、串口断连、雷达掉线
- 对策：先单独完成降压模块带载测试，再接系统

### 风险2：平衡底盘对高层控制不够宽容

- 表现：树莓派下发速度不平滑导致姿态波动
- 对策：高层只发有限幅、低频更新的目标速度，底层继续由 `STM32` 做闭环

### 风险3：`K210` 无法替代深度相机做深度任务

- 对策：明确职责切分
  - `K210` 负责识别与方向
  - 雷达负责距离与建图

### 风险4：旧 STM32 雷达代码与新 ROS2 雷达链路冲突

- 对策：不恢复旧雷达分支，统一走树莓派侧雷达主线

### 风险5：功能点很多，最后集成阶段爆炸

- 对策：每阶段都保留“阶段可演示版本”，不要等全部做完再合并

## 11. 当前建议的下一步

从今天开始，建议立即进入以下三个动作：

1. 先确认树莓派5供电方案与串口接线方案。
2. 在树莓派5上优先搭起 `Docker + ROS2` 最小环境。
3. 同时在 `STM32` 侧设计第一版“树莓派控制协议”，为后续联调做准备。

## 12. 结论

本项目后续所有开发都应围绕以下最终目标展开：

- 以当前平衡小车工程为基础；
- 融合 `STM32 + 树莓派5 + K210 + T-mini Plus`；
- 完成任务书中除机械臂外的核心功能；
- 用 `K210` 替代深度相机承担视觉任务；
- 最终交付一台可以独立完成现场演示的机器人小车。
