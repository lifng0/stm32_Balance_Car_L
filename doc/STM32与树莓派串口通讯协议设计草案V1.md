# STM32 与树莓派串口通讯协议设计草案 V1

更新时间：2026-06-02

## 1. 文档目的

这份文档用于定义当前项目中 `STM32` 与树莓派之间的第一版串口通讯协议，供审核与后续代码实现使用。

本协议的设计目标是：

- 先快速打通 `STM32 <-> 树莓派` 最小闭环
- 尽量贴合当前现有工程变量与模式定义
- 保证掉线、误码、异常命令时小车默认更安全
- 不为第一版引入不必要的复杂性

## 2. 适用范围

本协议只用于：

- 树莓派向 `STM32` 发送模式和运动控制命令
- `STM32` 向树莓派回传状态、应答和错误信息

本协议当前不负责：

- `K210` 原始识别数据全量透传
- 雷达原始点云传输
- 大文件传输
- 固件升级

## 3. 当前工程约束

结合现有代码，协议设计需要遵守这些现实约束：

- `K210` 当前固定使用 `USART2`
- `USART1` 当前保留为调试口
- `STM32 <-> 树莓派` 后续走 `UART3 / USART3`
- 树莓派物理接法默认为 `STM32 UART3 -> TTL转USB -> 树莓派 USB`
- 当前核心控制入口变量已经存在：
  - `Car_Mode mode`
  - `float Move_X`
  - `float Move_Z`
  - `u8 Stop_Flag`
  - `u8 lower_power_flag`

## 4. 设计原则

- 第一版只做二进制帧协议，不做文本协议
- 第一版优先实现“可控、可回显、可保护”
- 所有多字节字段统一使用小端序
- 树莓派是高层控制端，`STM32` 是底层执行端
- 若通讯失联，`STM32` 必须进入安全状态
- 命令必须可确认，避免“发了但不知道是否生效”

## 5. 串口参数建议

第一版建议使用：

- 波特率：`115200`
- 数据位：`8`
- 校验位：`None`
- 停止位：`1`

这样选的原因：

- 第一版调试最省心
- 对控制命令和状态回传已经足够
- 比起更高波特率，更有利于先把协议稳定跑通

如果后续发现带宽不足，再考虑升级到 `230400`

## 6. 帧格式

统一帧格式如下：

| 字段 | 长度(Byte) | 说明 |
|---|---:|---|
| SOF1 | 1 | 帧头1，固定 `0xAA` |
| SOF2 | 1 | 帧头2，固定 `0x55` |
| VER | 1 | 协议版本，V1 固定 `0x01` |
| CMD | 1 | 命令字 |
| SEQ | 1 | 帧序号，发送端循环递增 |
| LEN | 1 | 载荷长度 `Payload` 的字节数 |
| PAYLOAD | N | 载荷 |
| CHK | 1 | 校验字节 |

### 6.1 校验规则

第一版建议使用：

- `CHK = VER ^ CMD ^ SEQ ^ LEN ^ PAYLOAD[0] ^ ... ^ PAYLOAD[N-1]`

说明：

- 即从 `VER` 到 `PAYLOAD` 的逐字节异或
- 帧头 `0xAA 0x55` 不参与校验
- 第一版先用异或，代码量小，便于快速落地

## 7. 命令方向

### 树莓派 -> STM32

- 控制类命令
- 模式切换命令
- 心跳命令
- 查询命令

### STM32 -> 树莓派

- `ACK` 应答
- `STATUS` 状态回传
- `ERROR` 错误回传
- 心跳回包

## 8. 命令字定义

### 8.1 树莓派 -> STM32

| CMD | 名称 | 说明 |
|---|---|---|
| `0x01` | `PING` | 连通性测试 |
| `0x02` | `SET_ENABLE` | 启动/停止控制 |
| `0x03` | `SET_MODE` | 设置运行模式 |
| `0x04` | `SET_MOVE` | 设置 `Move_X / Move_Z` |
| `0x05` | `QUERY_STATUS` | 主动请求状态 |
| `0x06` | `HEARTBEAT` | 周期心跳 |
| `0x07` | `EMERGENCY_STOP` | 急停 |

### 8.2 STM32 -> 树莓派

| CMD | 名称 | 说明 |
|---|---|---|
| `0x81` | `ACK` | 成功应答 |
| `0x82` | `NACK` | 失败应答 |
| `0x83` | `STATUS` | 状态回传 |
| `0x84` | `EVENT` | 事件上报 |
| `0x85` | `HEARTBEAT_ACK` | 心跳回包 |

## 9. 载荷定义

## 9.1 `PING` `0x01`

载荷：

- 无

期望行为：

- `STM32` 收到后返回 `ACK`

## 9.2 `SET_ENABLE` `0x02`

载荷：

| 偏移 | 类型 | 含义 |
|---|---|---|
| 0 | `uint8` | `0=停止`，`1=允许控制` |

映射建议：

- `0` -> `Stop_Flag = 1`
- `1` -> `Stop_Flag = 0`

说明：

- 这是“总开关”命令
- 即使模式和速度目标已经设置，若 `Stop_Flag = 1`，电机仍不应输出

## 9.3 `SET_MODE` `0x03`

载荷：

| 偏移 | 类型 | 含义 |
|---|---|---|
| 0 | `uint8` | 模式编号 |

模式编号建议直接对齐当前工程 `Car_Mode` 枚举：

| 值 | 模式 |
|---:|---|
| `0` | `Normal` |
| `1` | `Weight_M` |
| `2` | `K210_QR` |
| `3` | `K210_Line` |
| `4` | `K210_Follow` |
| `5` | `K210_SelfLearn` |
| `6` | `K210_mnist` |

非法值处理建议：

- 超出 `0..6` 则返回 `NACK`
- 不修改当前模式

## 9.4 `SET_MOVE` `0x04`

载荷：

| 偏移 | 类型 | 含义 |
|---|---|---|
| 0..1 | `int16` | `Move_X`，单位建议 `0.1` |
| 2..3 | `int16` | `Move_Z`，单位建议 `0.1` |

换算建议：

- 实际值 `12.3` 对应发送 `123`
- `STM32` 收到后换算成：
  - `Move_X = raw_x / 10.0f`
  - `Move_Z = raw_z / 10.0f`

原因：

- 避免第一版直接发 `float`
- 整数协议更稳、更容易抓包排查

限幅建议：

- `STM32` 收到后再做一次本地限幅
- 第一版建议：
  - `Move_X` 限幅到 `[-30.0, 30.0]`
  - `Move_Z` 限幅到 `[-30.0, 30.0]`

说明：

- 这里的限幅值和当前 `K210` 跟随模式中的速度量级保持一致，便于后续统一

## 9.5 `QUERY_STATUS` `0x05`

载荷：

- 无

期望行为：

- `STM32` 立即回一帧 `STATUS`

## 9.6 `HEARTBEAT` `0x06`

载荷：

- 无

期望行为：

- `STM32` 返回 `HEARTBEAT_ACK`
- 同时刷新“通讯在线计时器”

## 9.7 `EMERGENCY_STOP` `0x07`

载荷：

- 无

期望行为：

- `STM32` 立即执行：
  - `Stop_Flag = 1`
  - `Move_X = 0`
  - `Move_Z = 0`
- 回 `ACK`

## 9.8 `ACK` `0x81`

载荷：

| 偏移 | 类型 | 含义 |
|---|---|---|
| 0 | `uint8` | 被确认的命令字 |
| 1 | `uint8` | 被确认的序号 `SEQ` |

## 9.9 `NACK` `0x82`

载荷：

| 偏移 | 类型 | 含义 |
|---|---|---|
| 0 | `uint8` | 被拒绝的命令字 |
| 1 | `uint8` | 被拒绝的序号 `SEQ` |
| 2 | `uint8` | 错误码 |

错误码建议：

| 值 | 含义 |
|---:|---|
| `1` | 校验错误 |
| `2` | 长度错误 |
| `3` | 非法命令 |
| `4` | 非法参数 |
| `5` | 当前状态不允许执行 |

## 9.10 `STATUS` `0x83`

第一版建议状态载荷如下：

| 偏移 | 类型 | 含义 |
|---|---|---|
| 0 | `uint8` | 当前模式 |
| 1 | `uint8` | `Stop_Flag` |
| 2 | `uint8` | `lower_power_flag` |
| 3..4 | `int16` | 当前 `Move_X`，单位 `0.1` |
| 5..6 | `int16` | 当前 `Move_Z`，单位 `0.1` |
| 7..8 | `uint16` | 电池电压，单位 `0.01V` |
| 9..10 | `int16` | 车体角度，单位 `0.1deg` |

说明：

- 电池电压例如 `12.34V` 发送为 `1234`
- 角度字段用于调试和安全监测，具体取哪个姿态变量后续实现时再最终确认

## 9.11 `EVENT` `0x84`

第一版建议只保留少量关键事件：

| 偏移 | 类型 | 含义 |
|---|---|---|
| 0 | `uint8` | 事件码 |

事件码建议：

| 值 | 含义 |
|---:|---|
| `1` | 低压进入保护 |
| `2` | 电压恢复 |
| `3` | 倾倒或被拿起，进入停止 |
| `4` | 通讯超时进入停止 |

## 9.12 `HEARTBEAT_ACK` `0x85`

载荷：

- 无

## 10. 心跳与失联保护

第一版建议：

- 树莓派每 `100ms` 发送一次 `HEARTBEAT`
- `STM32` 若连续 `500ms` 未收到心跳，则判定失联

失联后 `STM32` 处理建议：

- `Move_X = 0`
- `Move_Z = 0`
- `Stop_Flag = 1`
- 发送或缓存一个 `EVENT(4)`

说明：

- 对平衡车来说，失联后默认停车比继续保持旧命令更安全

## 11. 命令生效规则

为了避免控制冲突，第一版建议遵循下面规则：

- 只有树莓派接管模式下，树莓派发送的 `SET_MOVE` 才持续生效
- `SET_ENABLE=0` 或 `EMERGENCY_STOP` 优先级最高
- 若当前模式不适合接受外部 `Move_X / Move_Z`，可返回 `NACK`

这里有一个待审核点：

- 是让树莓派直接复用现有 `Normal` 模式
- 还是新增一个专门的 `Pi_Control` 模式

我的建议是：

- 第一版先复用 `Normal`
- 等链路跑通后，再评估是否需要新增独立模式

这样可以最小化对现有工程的侵入

## 12. 典型报文示例

## 12.1 例子：`PING`

发送：

- `AA 55 01 01 10 00 10`

说明：

- `VER=01`
- `CMD=01`
- `SEQ=10`
- `LEN=00`
- `CHK = 01 ^ 01 ^ 10 ^ 00 = 10`

## 12.2 例子：`SET_ENABLE(1)`

发送载荷：

- `01`

完整帧：

- `AA 55 01 02 11 01 01 13`

校验：

- `01 ^ 02 ^ 11 ^ 01 ^ 01 = 13`

## 12.3 例子：`SET_MOVE`

假设：

- `Move_X = 12.0`
- `Move_Z = -8.0`

则载荷原始值：

- `Move_X = 120 = 0x0078`
- `Move_Z = -80 = 0xFFB0`

小端载荷：

- `78 00 B0 FF`

## 13. 第一版实现建议

## 13.1 STM32 侧

- 新增 `USART3` 初始化
- 新增接收环形缓冲区
- 新增按字节解析的状态机
- 新增命令分发函数
- 新增状态打包函数
- 新增通讯超时计时器

## 13.2 树莓派侧

- 先写一个简单串口桥脚本
- 支持：
  - 打开串口
  - 发送 `PING`
  - 发送 `SET_ENABLE`
  - 发送 `SET_MODE`
  - 发送 `SET_MOVE`
  - 打印收到的 `ACK/STATUS/NACK`

## 14. 待审核问题

这份草案里，我建议你重点审核下面几点：

1. 第一版波特率是否固定为 `115200`
2. 是否复用 `Normal` 作为树莓派控制模式
3. `SET_MOVE` 的单位是否用 `0.1`
4. `Move_X / Move_Z` 限幅是否采用 `[-30.0, 30.0]`
5. 心跳周期 `100ms`、超时 `500ms` 是否合适
6. `STATUS` 中是否保留角度字段
7. 第一版是否需要加入更多状态字段

## 15. 建议结论

我的建议是：

- 先按这份协议的最小集落地
- 第一阶段只实现：
  - `PING`
  - `SET_ENABLE`
  - `SET_MODE`
  - `SET_MOVE`
  - `QUERY_STATUS`
  - `HEARTBEAT`
  - `ACK/NACK/STATUS`

这样最有利于尽快打通真实链路。
