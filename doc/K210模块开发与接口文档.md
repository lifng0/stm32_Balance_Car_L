## 1. 文档目的

这份文档用于给本项目后续的 `K210 / CanMV` 开发提供统一入口，目标是让开发者能清楚回答下面几件事：

- 这块 `K210` 板子有哪些现成硬件能力可以直接调用

- `K210` 如何和 `STM32`、树莓派现有系统对接
- 后续视觉算法代码建议放哪里、按什么方式部署

这份文档同时面向两类开发方式：

- 快速开发：尽量复用 Yahboom 官方封装和现成例程
- 自定义开发：直接用 `machine`、`lcd`、`sensor`、`fpioa_manager` 自己写底层逻辑

### 3.2 建议开发流程

推荐按这个顺序开发：

1. 先用 `CanMV IDE` 直接运行最小例程，确认板子和出厂固件正常
2. 再运行 `LCD`、`Camera`、`UART` 单功能例程，逐项确认
3. 单功能正常后，再写组合程序
4. 组合程序稳定后，再保存成 `main.py` 作为上电自启动脚本

### 3.3 上电运行与烧录的区别

开发时有两种常见方式：

- 在 `CanMV IDE` 里直接点击运行
  - 优点：最快速，适合调试
  - 缺点：断电不会保留
- 保存为 `main.py` 到板载文件系统
  - 优点：上电自动执行
  - 适合正式部署

建议：

- 联调阶段优先用 `IDE` 直接运行
- 逻辑确认稳定后再落成 `main.py`

### 4.2 摄像头

官方基础例程：

- `F:\BaiduNetdiskDownload\K210视觉模块\7.程序源码汇总\程序源码\03-Hardware\camera.py`
- `F:\BaiduNetdiskDownload\K210视觉模块\7.程序源码汇总\程序源码\01-Basics\helloworld.py`

```python
import sensor, lcd, time

lcd.init()

sensor.set_framesize(sensor.QVGA)
```

### 4.3 UART 串口通信

官方基础例程：

- `F:\BaiduNetdiskDownload\K210视觉模块\7.程序源码汇总\程序源码\03-Hardware\yb_uart.py`

这个例程非常关键，因为它直接说明了 Yahboom 当前这块板子的 `UART2` 引脚分配和常用初始化方式：

```python
from fpioa_manager import fm
from machine import UART

fm.register(6, fm.fpioa.UART2_RX)
fm.register(8, fm.fpioa.UART2_TX)
uart = UART(UART.UART2, 115200, 8, 0, 0, timeout=1000, read_buf_len=4096)
```

结论：

- `IO6 -> UART2_RX`
- `IO8 -> UART2_TX`
- 推荐参数：`115200 8N1`

这是当前项目和 `STM32 USART2` 联调时应当遵循的配置。

## 5. 低层接口和官方封装接口怎么选

Yahboom 资料里常见两种编程方式。

### 5.1 方式 A：直接用底层模块

典型导入：

```python
import lcd
import sensor
from machine import UART
from fpioa_manager import fm
```

优点：

- 透明
- 适合自己定义通信协议
- 适合和我们当前的 `STM32 / 树莓派` 系统做深度集成

缺点：

- 需要自己做引脚绑定、协议解析、异常处理

### 5.2 方式 B：使用 Yahboom 封装

典型导入：

```python
from modules import ybserial
from modules import ybrgb
from modules import ybkey
```

优点：

- 写起来快
- 很多官方扩展示例都直接这么用

缺点：

- 抽象层更厚
- 行为和协议细节不如底层直观
- 换板子、换固件时可移植性一般

### 5.3 当前项目建议

对于本项目，建议采用：

- 图像采集与显示：继续用 `sensor + lcd`
- 串口通信：优先自己用 `machine.UART + fm.register`
- 只有在快速验证官方示例时，才使用 `modules.ybserial`

原因很简单：

- 我们后面需要明确定义 `K210 <-> STM32` 或 `K210 <-> 树莓派` 的协议
- 自己控制 `UART2` 更稳，也更容易定位问题

## 6. 已确认的当前项目接线关系

### 6.1 K210 与树莓派

根据当前接线方案，`K210 UART2` 不再直连 `STM32 USART2`，而是改成：

- `K210 UART2 -> TTL转USB模块 -> 树莓派 USB`

`K210` 侧代码依据仍然是 Yahboom 官方 `UART2` 例程：

- `IO6 = UART2_RX`
- `IO8 = UART2_TX`

因此当前推荐接线为：

- `K210 IO8 (UART2_TX) -> TTL转USB RX`
- `K210 IO6 (UART2_RX) -> TTL转USB TX`
- `K210 GND -> TTL转USB GND`
- `TTL转USB USB -> 树莓派 USB`

### 6.2 当前项目里的 K210 测试脚本

当前仓库中的测试脚本：

- `F:\stm32_Balance_Car_L\tools\canmv_k210_uart_demo.py`

这份脚本的定位是：

- 验证 `LCD` 是否亮屏
- 验证 `UART2` 是否能正常接收来自 `STM32 / 树莓派` 的消息
- 出错时尽量把错误写到屏幕上

### 7.3 最小 UART2 文本接收模板

```python
import lcd
import time
from fpioa_manager import fm
from machine import UART

fm.register(6, fm.fpioa.UART2_RX, force=True)
fm.register(8, fm.fpioa.UART2_TX, force=True)
uart = UART(UART.UART2, 115200, 8, 0, 0, timeout=1000, read_buf_len=4096)

lcd.init()
lcd.clear(lcd.BLUE)
lcd.draw_string(20, 20, "UART2 ready", lcd.WHITE, lcd.BLUE)

while True:
    if uart.any():
        data = uart.read()
        if data:
            text = data.decode("utf-8", errors="replace")
            lcd.clear(lcd.BLUE)
            lcd.draw_string(20, 20, text, lcd.YELLOW, lcd.BLUE)
    time.sleep_ms(50)
```

用途：

- 判断 `K210 UART2` 是否真收到上位机数据

## 8. 官方通信示例里用到的协议思路

在下面这组官方扩展示例中：

- `F:\BaiduNetdiskDownload\K210视觉模块\6.拓展例程（另购）\6.2 多种主控通讯教程\6.配套K210从机程序\k210程序\K210-AI(stm32_pico_arduino)`

`K210` 不是简单发文本，而是发一类结构化数据帧。比如：

- `1_color_recognition_1.py`
- `3_find_qrcodes_3.py`

它们共同特征是：

- 帧头：`0x24`，也就是 `$`
- 帧尾：`0x23`，也就是 `#`
- 中间带长度、例程编号、数据组、CRC
- 坐标值按低字节在前的方式发

这说明官方推荐路线其实是：

- 简单联调用文本
- 真正传视觉结果时，使用结构化数据帧

### 8.1 对本项目的建议

当前项目建议分两阶段：

1. 联调阶段
   - 先用简单文本协议
   - 例如 `HELLO`、`MODE:QR`、`QR:abc123`
2. 正式阶段
   - 再切到结构化帧
   - 例如发送 `x / y / w / h / class / score`

这样能大幅降低早期排错成本。

## 9. 与当前小车系统的推荐集成方式

### 9.1 当前架构

当前系统建议架构是：

- `K210` 负责视觉识别
- `STM32` 负责底层执行和本地模式
- 树莓派负责高层逻辑和 `ROS`

因此当前现阶段最自然的链路是：

- `K210 -> 树莓派串口服务/ROS`
- `STM32 -> 树莓派串口服务/ROS`

如有必要，树莓派可以把 `K210` 的原始数据再次转发给 `STM32`，但这不再是默认主链路。

### 9.2 当前阶段推荐的数据内容

建议 `K210` 发给 `STM32` 的内容尽量简单：

- 文本类结果
  - 二维码字符串
  - 分类标签
  - 手写数字结果
- 框类结果
  - `x`
  - `y`
  - `w`
  - `h`
  - `score`

### 9.3 不建议 K210 直接做的事

当前项目里不建议让 `K210` 直接承担：

- 小车主运动控制闭环
- 雷达融合控制
- 高层状态管理

这些应继续由 `STM32` 或树莓派完成。

## 11. 推荐调试顺序

遇到任何问题，建议按这个顺序排：

1. 跑 `LCD` 例程，确认屏幕亮
2. 跑 `Camera` 例程，确认摄像头画面正常
3. 跑 `UART2` 最小接收例程，确认串口正常
4. 再运行组合脚本

不要一开始就：

- `LCD + Camera + UART + AI + STM32协议`

一起上，因为这样很难定位故障点。

## 12. 常见问题与建议

### 12.1 脚本烧进去后完全黑屏

优先排查：

- 脚本是否真的被执行
- 文件是否保存为 `main.py`
- 是否在顶层初始化阶段就异常退出
- `UART` 引脚或参数是否错误

建议先跑最小 `LCD` 自检模板。

### 12.2 LCD 能亮，但自定义 UART 脚本不起作用

优先确认：

- 是否按 `IO6=RX`、`IO8=TX`
- 是否用的是 `UART2`
- 参数是否为 `115200 8N1`
- 是否和 `STM32` 交叉接线
- 地线是否共地

### 12.3 官方示例能跑，自己脚本跑不起来

优先做法：

- 先复制官方最小例程
- 每次只改一处
- 不要一次性改引脚、协议、显示、算法

## 13. 本项目当前建议结论

如果只是为了尽快把 `K210` 接回当前小车系统，推荐这样做：

1. 先用 [canmv_k210_uart_demo.py](F:\stm32_Balance_Car_L\tools\canmv_k210_uart_demo.py) 打通 `LCD + UART2 + 树莓派串口`

4. 最后再把文本升级成结构化视觉结果

这样最稳，也最容易排查。
