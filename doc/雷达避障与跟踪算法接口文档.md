# 雷达避障与跟踪算法接口文档

更新时间：2026-06-03

## 1. 文档目的

本文面向“雷达避障与雷达跟踪”算法开发同学，目标是让你在不需要了解整套项目历史的情况下，直接开始编写可运行的程序，并在后续集成到当前小车系统中。

本文重点说明：

- 当前系统架构中，雷达数据从哪里来
- 算法程序应该读什么接口
- 算法程序怎样控制小车
- 算法最终应该部署到哪里
- 调试期和最终集成期分别怎样运行

## 2. 当前系统架构

当前项目采用两层结构：

- 宿主机控制层
  - `tools/pi_car_coordinator.py`
  - `tools/pi_serial_bridge.py`
  - `tools/tminiplus_bridge.py`
- Docker 容器内 ROS 2 功能层
  - `ros2_ws/src/balance_car_bridge`
  - `ros2_ws/src/balance_car_lidar`
  - `ros2_ws/src/balance_car_behavior`
  - `ros2_ws/src/balance_car_perception`

当前雷达接入路径是：

`T-mini Plus -> 树莓派宿主机 -> pi_car_coordinator -> localhost TCP backend -> ROS 2 节点`

这意味着：

- 算法同学**不要直接抢占雷达串口**
- 算法同学优先通过 ROS 话题或本地 backend 读取雷达结果

## 3. 当前可直接使用的数据接口

### 3.1 推荐主接口：`/lidar/summary_json`

来源：

- 节点文件：[lidar_summary_node.py](F:\stm32_Balance_Car_L\ros2_ws\src\balance_car_lidar\balance_car_lidar\lidar_summary_node.py)

话题类型：

- `std_msgs/msg/String`

消息内容：

- JSON 字符串

当前字段格式：

```json
{
  "points": 430,
  "scan_frequency_hz": 10.0,
  "scan_time_s": 0.0096,
  "front_min_distance_m": 1.164,
  "front_left_min_distance_m": 0.544,
  "front_right_min_distance_m": 0.812,
  "closest_target_angle_deg": 55.7,
  "closest_target_distance_m": 0.544,
  "device": "/dev/serial/by-id/...",
  "scan_ok": true
}
```

字段含义：

- `front_min_distance_m`
  - 正前方扇区最小距离，当前扇区约为 `-15° ~ +15°`
- `front_left_min_distance_m`
  - 左前扇区最小距离，当前扇区约为 `+15° ~ +60°`
- `front_right_min_distance_m`
  - 右前扇区最小距离，当前扇区约为 `-60° ~ -15°`
- `closest_target_angle_deg`
  - 前方候选区域内最近目标角度
- `closest_target_distance_m`
  - 前方候选区域内最近目标距离
- `scan_ok`
  - 当前扫描是否有效

这个接口已经足够完成：

- 最小避障
- 最小跟踪
- 低速演示

### 3.2 推荐辅助接口：`/car/system_state_json`

来源：

- 节点文件：[bridge_node.py](F:\stm32_Balance_Car_L\ros2_ws\src\balance_car_bridge\balance_car_bridge\bridge_node.py)

话题类型：

- `std_msgs/msg/String`

消息内容示例：

```json
{
  "system_mode": "running",
  "lidar_enabled": true,
  "lidar_required": true,
  "pi_ready": true,
  "lidar_ready": true,
  "system_ready": true,
  "paused_by_pickup": false,
  "shutdown_started": false,
  "backend_port": 8765
}
```

建议用途：

- 判断雷达是否已就绪
- 判断当前是否处于暂停态
- 判断系统是否允许运行算法

### 3.3 推荐辅助接口：`/car/state_json`

话题类型：

- `std_msgs/msg/String`

主要字段：

```json
{
  "mode": 0,
  "mode_name": "Normal",
  "stop_flag": 1,
  "low_power": 0,
  "move_x": 0.0,
  "move_z": 0.0,
  "battery": 12.0,
  "angle": 0.5
}
```

建议用途：

- 读取当前模式
- 读取当前底盘已经执行的目标量
- 监视电池和姿态

## 5. 控制小车的方法

这一节分成两部分：

- 调试期最快可用的方法
- 最终集成期推荐方法

### 5.1 调试期最快可用方法

当前最快的控制方式是直接调用：

- [pi_serial_bridge.py](F:\stm32_Balance_Car_L\tools\pi_serial_bridge.py)

它支持的核心命令是：

- `enable 0|1`
- `mode N`
- `move MOVE_X MOVE_Z`
- `stop`
- `status`

命令含义：

- `enable 1`
  - 允许小车动作，等价于 `Stop_Flag = 0`
- `enable 0`
  - 禁止小车动作，等价于 `Stop_Flag = 1`
- `move x z`
  - 设置控制目标
- `stop`
  - 急停

注意：

- `move_x` 和 `move_z` 当前不是严格 ROS 语义下的 `m/s` 和 `rad/s`
- 它们本质上是底盘控制量，建议先按经验量级使用
- 当前协议内部会限幅到 `[-30.0, 30.0]`

推荐起步范围：

- `move_x`
  - `-15 ~ +15`
- `move_z`
  - `-400 ~ +400` 是旧 `K210` 逻辑常见量级
  - 但通过 `pi_serial_bridge.py move` 发送时，建议先从 `-15 ~ +15` 的小量级开始摸底

原因：

- 目前 `STM32` 的不同模式内部对这些控制量的解释并不完全一致
- 算法调试时先保守，避免第一次就把车打飞

#### 调试期重要约束

如果你直接用 `pi_serial_bridge.py` 控车，**必须避免和 `pi_car_coordinator` 同时抢占串口**。

也就是说，下面两种方式只能选一种：

1. 独立调试模式
   - 停掉 `balance-car-coordinator.service`
   - 你的算法程序直接调用 `pi_serial_bridge.py`
2. 正式集成模式
   - 保持 `balance-car-coordinator.service` 运行
   - 通过 backend 控制适配层下发动作

### 5.2 最终集成期推荐方法

最终部署时，**不建议算法节点直接打开 STM32 串口**。

推荐方法是：

- 算法节点部署在 ROS 2 容器内
- 算法节点通过 `localhost TCP backend` 和 `pi_car_coordinator` 交互
- `pi_car_coordinator` 仍然是唯一串口持有者

当前已经存在的 backend 读取命令：

- `{"cmd": "get_state"}`
- `{"cmd": "get_vision"}`

当前已经落地的控制 backend 命令：

- `{"cmd": "set_enable", "value": 0|1}`
- `{"cmd": "set_mode", "value": 0..6}`
- `{"cmd": "set_move", "move_x": 0.0, "move_z": 0.0}`
- `{"cmd": "stop"}`

这些命令由树莓派宿主机上的 `pi_car_coordinator.py` 统一持有串口并转发，算法本身不应直接依赖串口细节。

因此建议算法同学按下面方式开发：

1. 先把算法逻辑写成纯函数
2. 用“控制适配器”输出动作
3. 调试期让控制适配器调用 `pi_serial_bridge.py`
4. 正式集成时，把控制适配器替换成 backend 调用

### 6.3 控制适配器建议

建议单独写一个：

- `controller_adapter.py`

职责：

- 提供统一方法：
  - `set_enable(True/False)`
  - `set_mode(mode_id)`
  - `set_move(move_x, move_z)`
  - `stop()`
- 在调试阶段走 `pi_serial_bridge.py`
- 在正式集成阶段切换成 backend 控制命令

这样算法同学不用重写核心逻辑。

### 6.4 当前已经提供的模板节点

当前仓库里已经有两份可直接运行的模板：

- [lidar_avoid_node.py](F:\stm32_Balance_Car_L\ros2_ws\src\balance_car_navigation\balance_car_navigation\lidar_avoid_node.py)
- [lidar_follow_node.py](F:\stm32_Balance_Car_L\ros2_ws\src\balance_car_navigation\balance_car_navigation\lidar_follow_node.py)

它们的共同特点是：

- 继续使用本文前面定义的数据接口
  - 读 `/lidar/summary_json`
  - 读 `/car/system_state_json`
- 控制不直接碰串口
  - 统一走 `controller_adapter.py`
  - 由树莓派上的 `pi_car_coordinator` 持有串口并转发
- 都会在首次真正下发动作前调用一次 `set_enable(True)`
- 如果系统未 ready、被拿起暂停、或者雷达扫描无效，都会自动发 `set_move(0.0, 0.0)`

对算法同学来说，最重要的结论是：

- 你不需要先搭通信链
- 你主要改“决策逻辑”和“参数”
- 如果想做新的策略，优先复制其中一个节点再改，而不是从零重新接线

## 7. 建议的避障算法

### 7.1 最小可演示版本

输入：

- `front_min_distance_m`
- `front_left_min_distance_m`
- `front_right_min_distance_m`

建议阈值起步值：

- `warning_distance = 0.80`
- `stop_distance = 0.45`
- `turn_clear_distance = 0.70`

建议行为：

1. `front > warning_distance`
   - 前进
2. `stop_distance < front <= warning_distance`
   - 减速前进
3. `front <= stop_distance`
   - 停止前进
   - 比较左右哪边更空
   - 向更空的一边给转向

建议输出例子：

- 正常前进
  - `set_enable(True)`
  - `set_move(8.0, 0.0)`
- 减速
  - `set_move(4.0, 0.0)`
- 左转避障
  - `set_move(0.0, -8.0)`
- 右转避障
  - `set_move(0.0, 8.0)`

### 7.2 当前模板节点参数

`lidar_avoid_node.py` 当前内置参数如下：

- `backend_host = 127.0.0.1`
- `backend_port = 8765`
- `warning_distance = 0.80`
- `stop_distance = 0.45`
- `turn_clear_distance = 0.70`
- `forward_speed = 8.0`
- `slow_speed = 4.0`
- `turn_speed = 8.0`

推荐理解：

- `warning_distance`
  - 开始减速的距离
- `stop_distance`
  - 不能继续直行的距离
- `turn_clear_distance`
  - 判断“哪边更安全”时的参考距离
- `forward_speed / slow_speed / turn_speed`
  - 都是发给底盘的控制量，不是严格物理单位

建议组员优先调的只有两类：

- 距离阈值
- 前进/转向速度

不建议一开始就动的部分：

- backend 地址
- 控制适配器接口形式

## 8. 建议的跟踪算法

### 8.1 最小可演示版本

输入：

- `closest_target_angle_deg`
- `closest_target_distance_m`

建议阈值起步值：

- `track_angle_deadband_deg = 8`
- `target_distance_m = 0.8`
- `distance_deadband_m = 0.15`

建议行为：

1. 如果没有目标
   - `set_move(0.0, 0.0)`
2. 如果目标角度偏左
   - 左转对准
3. 如果目标角度偏右
   - 右转对准
4. 对准后根据距离前进或后退

建议输出例子：

- 目标在左侧
  - `set_move(0.0, -6.0)`
- 目标在右侧
  - `set_move(0.0, 6.0)`
- 目标太远
  - `set_move(6.0, 0.0)`
- 目标太近
  - `set_move(-4.0, 0.0)`

### 8.2 当前模板节点参数

`lidar_follow_node.py` 当前内置参数如下：

- `backend_host = 127.0.0.1`
- `backend_port = 8765`
- `target_distance = 0.70`
- `distance_tolerance = 0.12`
- `max_forward_speed = 8.0`
- `max_turn_speed = 8.0`
- `turn_gain = 0.18`
- `forward_gain = 12.0`
- `max_track_angle_deg = 60.0`

推荐理解：

- `target_distance`
  - 希望跟随时保持的目标距离
- `distance_tolerance`
  - 在这个误差范围内，不再前后移动
- `turn_gain`
  - 目标角度偏差换算成转向控制量的比例
- `forward_gain`
  - 目标距离误差换算成前进/后退控制量的比例
- `max_track_angle_deg`
  - 超过这个角度直接认为目标不适合继续跟

建议组员优先调的只有三项：

- `target_distance`
- `turn_gain`
- `forward_gain`

## 9. 推荐代码骨架

### 9.1 读取雷达摘要

```python
import json

from rclpy.node import Node
from std_msgs.msg import String


class DemoNode(Node):
    def __init__(self):
        super().__init__("demo_node")
        self.latest_lidar = None
        self.create_subscription(String, "/lidar/summary_json", self.on_lidar, 10)

    def on_lidar(self, msg: String):
        try:
            self.latest_lidar = json.loads(msg.data or "{}")
        except json.JSONDecodeError:
            self.latest_lidar = None
```

### 9.2 调试期控制适配器

```python
import subprocess


STM32_DEVICE = "/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0"
BRIDGE = "/home/lifng0/workspace/balance_car/scripts/pi_serial_bridge.py"


def run_bridge(*args):
    cmd = ["python3", BRIDGE, "--device", STM32_DEVICE, *map(str, args)]
    return subprocess.run(cmd, check=True, capture_output=True, text=True)


def set_enable(value: bool):
    run_bridge("enable", "1" if value else "0")


def set_move(move_x: float, move_z: float):
    run_bridge("move", move_x, move_z)


def stop():
    run_bridge("stop")
```

重要提醒：

- 这套写法只适合**调试期**
- 使用前应停掉 `balance-car-coordinator.service`

### 10.6 组员真正需要改哪些地方

如果组员是“写算法的人”，通常只需要改下面这些内容：

1. 节点里的决策逻辑
   - 例如 `if front > warning_distance` 这一类规则
2. 参数默认值
   - 例如 `warning_distance`、`turn_gain`
3. 必要时新增自己的策略节点
   - 例如复制一份 `lidar_avoid_node.py` 改成 `lidar_wall_follow_node.py`

通常不需要改的内容：

- `controller_adapter.py`
- backend TCP 协议
- `pi_car_coordinator.py`
- `bridge_node.py`

这部分不改的好处是：

- 算法逻辑和通信细节解耦
- 出问题时更容易定位
- 不会因为改算法把整条控制链打坏

不要把最终算法放在：

- `tools/`
- `STM32` 固件里
- `balance_car_lidar` 包内部

原因：

- 算法属于高层任务逻辑，不属于驱动层
