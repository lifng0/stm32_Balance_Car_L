# 现有系统与新增配件接线方案

更新时间：2026-05-30

## 1. 文档范围

这份文档只描述“当前旧系统”与“这次新增配件”之间需要新增或确认的接线关系，不重复展开小车内部已经完成的原有接线。

当前按以下边界执行：

- `K210` 继续接 `STM32 USART2`
- `USART1` 继续作为 `STM32` 调试口
- 原来给雷达预留的 `UART3 / USART3` 改作 `STM32 <-> TTL转USB模块 <-> 树莓派5 USB` 通信链路
- `T-mini Plus` 不再接 `STM32`，而是直接接树莓派5

## 2. 串口资源结论

### `STM32 USART1`

- 用途：调试口、日志口
- 说明：保持默认用途，不挪作树莓派主通信

### `STM32 USART2`

- 用途：`K210` 通信口
- 代码依据：当前工程中 `USART2` 初始化位于 [bsp_usart2.c](F:\stm32_Balance_Car_L\BSP\Usart1\bsp_usart2.c)
- 现状：继续沿用，不改接线

### `STM32 UART3 / USART3`

- 用途：改作 `STM32 <-> TTL转USB模块 <-> 树莓派5` 主通信口
- 说明：开发板正面图显示板上预留有 `UART3` 接口，后续实际接线时以板上丝印为最终准

## 3. 需要新增的接线

### 3.1 `STM32 <-> TTL转USB模块 <-> 树莓派5`

推荐把 `STM32 UART3 / USART3` 先接到一个 `3.3V TTL转USB` 模块，再由模块插到树莓派 `USB` 口：

- `STM32 TX3 -> TTL转USB模块 RX`
- `STM32 RX3 -> TTL转USB模块 TX`
- `STM32 GND -> TTL转USB模块 GND`
- `TTL转USB模块 USB -> 树莓派 USB`

补充说明：

- 推荐选择 `3.3V TTL` 逻辑电平的 `CP2102 / CH340 / FT232` 一类模块
- 这条链路本质上仍然是串口通信，只是树莓派侧会把它识别成一个 `USB` 串口设备
- `UART3` 口上如果同时带有 `5V` 引脚，这个 `5V` 不用于给 `TTL转USB` 模块反向供电，也不用于给树莓派供电
- 树莓派必须由独立降压稳压模块供电，不从 `STM32` 板载接口“顺手取电”
- 这种接法通常比树莓派 `GPIO` 杜邦线更稳固，更适合长期演示和搬动场景

### 3.2 电池 -> 降压稳压模块 -> 树莓派5

推荐按独立供电链路接线：

- 小车主电池输出 -> 降压稳压模块输入
- 降压稳压模块输出 `5V` -> 树莓派5 电源输入
- 降压稳压模块输出地 -> 树莓派地

设计要求：

- 输出电压目标固定为稳定 `5V`
- 供电能力要覆盖树莓派5实际运行电流，并预留雷达、USB外设、峰值负载余量
- 上电后优先验证树莓派在高负载下不会欠压重启

## 4. `T-mini Plus` 激光雷达接线方案

## 4.1 手册确认到的硬件约束

根据 [YDLIDAR T-mini Plus 数据手册_V1.0 (231214).pdf](F:\stm32_Balance_Car_L\doc\硬件资料\TminiPlus资料\YDLIDAR T-mini Plus 数据手册_V1.0 (231214).pdf) 与使用手册：

- 雷达接口为 `GH1.25-4P`
- 引脚为 `VCC / Tx / GND / Rx`
- 供电电压范围为 `4.8V-5.2V`
- 推荐供电为 `5V 1A`
- 通信接口为 `3.3V TTL UART`
- 默认串口参数为 `230400 8N1`

这意味着：

- 雷达逻辑电平与树莓派 `GPIO UART` 兼容
- 但雷达供电是 `5V`，不能直接从随意的弱电源口取电
- 如果走树莓派 `GPIO UART`，必须特别注意电源完整性和串口抗干扰

## 4.2 方案A：推荐方案

推荐采用：

- `T-mini Plus -> USB 转 UART 转接板 -> 树莓派 USB`

具体接法：

- 雷达 `VCC -> 转接板供电正端`
- 雷达 `GND -> 转接板 GND`
- 雷达 `Tx -> 转接板 RX`
- 雷达 `Rx -> 转接板 TX`
- 转接板 `USB_DATA` 口 -> 树莓派 USB 口

如果使用的是官方配套转接板，可按使用手册中的两个接口理解：

- `USB_DATA`：数据与供电复用，绝大多数情况下优先使用这个口
- `USB_PWR`：辅助供电口，当主机 `USB` 供电能力偏弱时再补充使用

推荐这套方案的原因：

- 接线最简单
- 与官方调试方式一致，问题更容易定位
- 不占用树莓派主串口
- 后续在树莓派上跑雷达驱动、`ROS2` 节点、`slam` 更直接

## 4.3 方案B：可用但不作为默认方案

备选方案为：

- `T-mini Plus TTL UART -> 树莓派 GPIO UART`

具体接法：

- 雷达 `Tx -> 树莓派 RX`
- 雷达 `Rx -> 树莓派 TX`
- 雷达 `GND -> 树莓派 GND`
- 雷达 `VCC -> 稳定 5V 电源`

这套方案能工作，但不作为首选，原因是：

- 需要额外确认树莓派串口口令行占用、串口使能、设备名固定等问题
- 雷达波特率较高，后期更容易遇到 GPIO 串口稳定性与布线干扰问题
- 树莓派主串口资源最好优先留给系统调试或其他特殊用途

只有在没有合适 `USB 转 UART` 转接板时，才建议临时采用这套方式。

## 5. 推荐的最终落地接线

为了缩短开发周期，建议最终按下面这套方案落地：

- `K210 <-> STM32 USART2`
- `STM32 USART1 -> 调试口`
- `STM32 UART3 / USART3 -> TTL转USB模块 -> 树莓派5 USB`
- `T-mini Plus -> USB 转 UART 转接板 -> 树莓派 USB`
- `电池 -> 降压稳压模块 -> 树莓派5`

这样分工最清晰：

- `STM32` 负责平衡控制与底层执行
- 树莓派5负责高层逻辑与 `ROS2`
- `K210` 负责视觉识别
- `T-mini Plus` 负责雷达感知与建图

## 6. 联调顺序建议

建议按下面顺序实际接线和上电：

1. 先确认降压稳压模块输出稳定 `5V`
2. 单独给树莓派5供电并完成基础启动测试
3. 接通 `STM32 -> TTL转USB模块 -> 树莓派5` 串口
4. 接通 `K210 <-> STM32`
5. 最后再接 `T-mini Plus -> 树莓派`

这样做可以把供电问题、串口问题、雷达驱动问题逐步拆开定位，避免一次性全接后难以判断故障来源。

## 7. 接好线后如何完成通信

最小通信流程按下面做即可：

1. `STM32` 侧继续使用现有 `UART3 / USART3` 作为与上位机通信的串口。
2. 把 `TTL转USB` 模块插到树莓派后，在树莓派上确认系统识别到了新的串口设备。
3. 树莓派程序打开对应串口设备，串口参数设置为与 `STM32` 完全一致。
4. 先做最小收发测试：
   - 树莓派发送一条简单命令帧
   - `STM32` 收到后返回状态帧或应答帧
5. 收发正常后，再接入正式协议：
   - 树莓派下发模式切换、速度、转向、启停命令
   - `STM32` 回传姿态、速度、电池、模式、故障状态

树莓派侧建议这样实现：

- 不要写死 `/dev/ttyUSB0`
- 优先使用 `/dev/serial/by-id/` 下的稳定设备路径
- 给 `STM32` 和雷达各自固定一条串口路径，避免插拔顺序变化后串口号漂移

通信内容建议保持最小化：

- 树莓派 -> `STM32`：模式、`Move_X`、`Move_Z`、启停
- `STM32` -> 树莓派：姿态、速度、电池、当前模式、错误状态

这样做的好处是：

- `STM32` 闭环控制逻辑不用变
- 树莓派只负责高层决策和命令发送
- 即使后续再换通信介质，串口协议本身也可以继续复用
